## 1. 한눈에 보기

**한 문장 정의**: 사용자가 프로젝트를 만들고 대화·문서·작업 디렉터리·Git 저장소를 연결하면, 시스템이 작업 변화를 자동 감지·분석하여 **현재 상태·변경사항·실제 진척도·다음 행동**을 시각적으로 보여주는 범용 AI 생산성 플랫폼.

**LINK AI가 *아닌* 것**: 단순 메모앱 · 문서저장소 · 파일탐색기 · 전체 지식그래프 뷰어 · 일반 RAG 챗봇 · 단순 PM 프로그램 · 파일 수정 횟수 집계기.

**세 가지 헤드라인 가치**:

| # | 가치 | 한 줄 |
| --- | --- | --- |
| 1 | **자동 변화 감지** | 실제 디렉터리·기록에서 무엇이 바뀌었는지 시스템이 먼저 잡는다 |
| 2 | **증거 기반 진척도** | 활동량(커밋 수·작업 시간)이 아니라 완료 조건 + 검증된 증거로 실제 진척을 계산 |
| 3 | **행동으로 연결되는 시각화** | 현재 상태·변화·병목·다음 행동을 보여주고 그 화면에서 바로 승인·업무 생성 |

## 2. 왜 만드나 — 문제

한 프로젝트를 하며 우리는 ChatGPT/Claude, 메신저, 노션, 로컬 폴더, Git, 칸반을 동시에 쓴다. 그 결과:

- **정보 분산** — 같은 프로젝트 자료가 여러 서비스·디렉터리에 흩어짐
- **오래된 정보 혼재** — 초기 계획·중간 변경·최신 결정이 다 검색되어 뭐가 유효한지 모름
- **생산성 도구의 역설** — 정리하려고 사용자가 태그·폴더·링크를 *계속* 관리해야 함
- **활동량 ≠ 진척도** — 커밋 수·작업 시간은 바쁨을 보여줄 뿐 완료를 증명하지 못함
- **가시성 부족** — 현재 상태·변경 원인·병목·우선순위가 한 화면에 없음
- **AI 비용·의존성** — 모든 걸 외부 LLM에 던지면 비용·속도·보안·장애의존이 커짐

→ "지금 어떤 상태? 최근 뭐 바뀜? 어떤 결정이 유효? 실제 어디까지? 뭐가 막고 있나? 지금 뭘 먼저? 근거 파일은?"에 즉답하지 못한다. LINK AI는 이를 **현재 상태 중심의 자동 구조화·시각화**로 푼다.

## 3. 핵심 개념

- **Project Digital Twin** — 현실 프로젝트(대화/문서/코드/파일/Git커밋/체크리스트)를 내부 모델(현재 목표/결정/결과물 상태/마일스톤/변경 이력/진척도/위험/다음 행동)로 동기화.
- **Current State** — "지금 유효한 것"만 기본 표시. 과거는 지우지 않고 이력으로 보존.
- **Evidence-based Progress** — `마일스톤 → 완료 조건 → 증거 → 검증 → 진척 반영`. **진척 %는 코드가 가중치로 계산하고, LLM은 임의 %를 정하지 않는다.**
- **Activity vs Progress** — 활동량(수정 파일 수·커밋 수·작업 시간)과 진척(완료 조건 충족·검증된 결과물·승인된 결정)을 **반드시 분리 표시**. 바쁜데 안 끝나는 착시(velocity theater)를 정면으로 겨냥.
- **Freshness 상태값** — `CURRENT` · `PENDING` · `CONFLICT` · `SUPERSEDED` · `STALE` · `ARCHIVED` · `INVALID`.

## 4. 어떻게 동작하나 — 파이프라인

```text
변경 이벤트 발생
  → 콘텐츠 해시 확인 ─(동일 해시)→ 종료   ※ 동일 파일 재분석 안 함
  → 이전 버전 조회
  → Diff 생성
  → 규칙 분석 (코드)
  → 필요 시에만 LLM 분석            ※ 애매한 것만
  → 현재 상태 / 변경 이력 / 마일스톤 증거에 반영
  → 확신 낮거나 중요하면 검토함으로 → 사용자 승인
  → NOW / MAP / CHANGES / TASKS 자동 갱신
```

**LLM 호출 라우팅** — 외부 API는 최종 fallback으로만:

```text
1. Cache  →  2. Rule Engine  →  3. Local Model(Ollama)  →  4. External API  →  5. Human Review
```

- **LLM 없이(코드로)**: 파일 변경 감지, 해시, 버전 비교, 날짜 파싱, 상태 머신, 체크리스트 집계, 진척도 계산, 화면 필터, 대시보드 집계, 권한, 캐시.
- **LLM이 담당**: 자유형 회의록 구조화, 사실·결정·업무 후보 추출, 의미적으로 같은 업무 판단, 의미적 충돌 분석, 변경 영향 설명, 자연어 질의 해석.

## 6. 화면

| 화면 | 역할 |
| --- | --- |
| **NOW** | 지금 가장 중요한 상태 — 진행 중 프로젝트, 최근 변경, 확인 필요, 마감 임박, 지연 업무, 다음 행동, 활동량 vs 진척도 |
| **MAP** | 현재 프로젝트 관계만(최대 20~30 노드) — 프로젝트/목표/결정/업무/사람/문서/일정/위험/결과물 |
| **CHANGES** | 시간 순 변경 — 변경 전/후, 일시, 파일, 이유, 영향받는 업무, 승인 상태 |
| **TASKS** | 상태별 업무 — 담당자, 우선순위, 마감일, 관련 증거, 지연 여부 |
| **ASK** | 자연어 탐색 — "이번 주 변경된 결정만", "이 결정에 영향받는 파일" → 답변 + 근거 + 실행 버튼 |

## 11. 엔지니어링 분석

> 이 절은 기획서의 요약이 아니라, **기획을 코드로 옮기기 전에 짚어야 할 판단**이다. 아키텍처·프로덕트·AI/LLM비용·보안·딜리버리·레드팀 **6개 전문 렌즈**가 기획서 v2.0을 각각 독립 분석했고, 그 결과를 종합했다.
>
> **읽는 법**: 여러 렌즈가 *독립적으로 겹쳐* 지적한 항목 = 가장 확실한 신호. 아래 🔴 표는 그 교집합이다.

### 11.1 총평

기획의 **골격은 이 도메인에 정확히 맞다.** 특히 세 가지 설계 결정이 뛰어나다 — (1) LLM을 최후의 fallback으로 미는 라우팅, (2) 원본/AI해석 분리 + 후보→승인 파이프라인, (3) 진척도 계산의 결정권을 코드로 회수. 대부분의 "AI 생산성" 제품이 전부 LLM에 던지는 것과 대비되는 실질적 차별점이다.

그러나 **제품의 존재 이유("정확한 현재 상태·진척 재구성")를 떠받치는 세 기둥이 기획서에서 가장 얇다.** 이 셋은 나중에 리팩터로 덧대기 어려운 기초 골조라, 미정 상태로 진행하면 헤드라인 가치가 조용히 무너진다. 6개 렌즈 전원이 **"조건부 진행(Go, with preconditions)"** 으로 수렴했다 — 방향은 옳으니 착수하되, 아래 🔴 세 가지는 코드 이전에 확정하라는 것.

### 11.2 강점 (설계가 이미 잘한 것)

- **LLM 경계선을 정확히 그음** — 진척도 숫자를 코드가 계산하고 LLM의 임의 %를 금지(§9.7). 신뢰성에서 가장 위험한 "숫자를 LLM에 맡기기"를 원천 차단.
- **버전 축을 캐시 키에 명시** — 문서분석 캐시 키에 `parserVersion·promptVersion·modelVersion`을 넣어(§11.1), 모델·프롬프트 교체 시 stale 결과가 남는 흔한 함정을 구조적으로 차단.
- **원본 불변 + 후보→검증→반영** — `ExtractedItem.confidence/status`, `Evidence.verification_status`(§14)가 "AI는 후보만 낸다"를 스키마로 지지. LLM 오판이 곧바로 데이터 오염이 되지 않음.
- **비용·속도 방어를 원칙으로 못박음** — 5.4/5.6/10.3이 LLM 남용으로 비용이 폭발하는 전형적 실패를 설계 단계에서 차단. MVP에서 LLM 의존 기능을 통째로 미뤄도 제품이 성립하게 해주는 지렛대.

### 11.3 🔴 착수 전 반드시 확정 (여러 렌즈 교집합 = 최고 신뢰)

| # | 문제 | 지적한 렌즈 | 왜 지금 확정해야 하나 | 방향 |
| --- | --- | --- | --- | --- |
| 🔴1 | **stateVersion 입도** — `state_version`이 Workspace에만 있는데(§14) QA·대시보드 캐시 키가 이걸 씀(§11.3~11.4). 한 프로젝트의 사소한 변경 하나가 워크스페이스 전체 캐시를 무효화 | 아키텍처 · AI/LLM · 레드팀 | 활동 많은 프로젝트일수록(제품이 노리는 바로 그 상황) 캐시 히트율이 붕괴 → **§22.4 비용 절감 전제가 파손**. 캐시가 stale은 막지만 비용 레버로선 무력화 | `project.state_version` 신설 + 무효화 매트릭스(어떤 트리거가 무엇을 bump하는지)를 표로 확정. 대시보드 키는 소속 프로젝트 버전들의 해시로 |
| 🔴2 | **로컬 에이전트 신뢰경계** — 해시·Diff·민감정보 마스킹이 전부 클라이언트에서 수행되고(§12·§20) 서버가 무검증 신뢰. 에이전트 인증·멱등키·서버측 재검증 부재. 게다가 `raw_content`가 서버 DB에 원문 전량 저장(§14) → "Local Only"(§20.2)는 **외부 LLM 미전송일 뿐 원문은 SaaS 서버에 적재** = 오명명 | 아키텍처 · 보안 · 딜리버리 · 레드팀 | 에이전트가 버그·변조되면 (a) 잘못된 값으로 현재 상태 오염, (b) 마스킹 건너뛴 시크릿이 그대로 외부 LLM행. 보안의 마지막 방어선을 신뢰 못 할 경계에 둠. DB 한 번 유출 = 사용자 로컬 전체 유출 | ① 외부 LLM 호출 직전 **서버측 재해시·2차 시크릿 필터** 강제(방어 이중화) ② 에이전트 스코프 토큰 + 요청 서명 + `POST /api/agent/events` 멱등키 ③ Local Only는 서버에 해시·파생상태만·원문은 로컬 유지 ④ 제외목록을 deny-list→**allow-list**로 + gitleaks류 파일내부 스캐너 내장 |
| 🔴3 | **변경 이벤트의 순서·멱등·트랜잭션 경계** — debounce(§9.3)와 OS WatchService coalescing이 겹쳐 이벤트가 병합·재배열됨. per-source 직렬화·순서 보장·멱등키·낙관적 락·아웃박스 부재 | 아키텍처 · 딜리버리 · 레드팀 | 이벤트 유실·재배열·부분실패가 누적되면 **버전 체인이 깨지고 Current State가 실제와 조용히 발산**. 이게 곧 "정확한 상태 재구성"이라는 제품의 심장 | Source별 단조 시퀀스로 서버가 순서 확정, `SourceVersion`에 낙관적 락, `ChangeEvent`를 상태머신(PENDING/PROCESSING/DONE/FAILED)+아웃박스+DLQ로. 각 단계 멱등화. **주기적 재대조(reconciliation) 배치**도 사실상 필수 |

### 11.4 가장 어려운 기술 난제 — "증거 → 완료조건 매핑"

**이것이 제품의 진짜 심장이자 최약점이다.** (프로덕트·AI/LLM·딜리버리·레드팀 4개 렌즈가 동일 지목)

- §9.7은 "진척 %를 코드로 정확 계산"이라 하지만, 코드가 하는 일은 이미 확정된 가중치를 더하는 **산수뿐**이다. 진짜 어려운 단계는 *"이 커밋/실행화면이 완료조건('주요 클릭 동작 확인')을 실제로 충족했는가"* 라는 **정답 라벨 없는 의미 판정** — 코드가 아니라 LLM/사람이다. "코드로 정확"이라는 표현이 근거 없는 확신을 만든다.
- 이 매핑 정확도가 체감 90% 아래면 사용자가 전부 재검증해야 하고, 그러면 **"사용자는 승인만"이라는 핵심 약속(§1.3)이 깨진다.**
- 파생 난제: **의미적 dedup/충돌 판정**(§10.2)은 비결정적이고 O(n²)이며 정답셋이 없어, 오탐이 검토함을 알림 홍수로 만들어 사용자가 검토함 자체를 무시(alert fatigue)하게 함.

**대응**: (1) 진척도를 단일 %가 아니라 **"완료조건 7개 중 4개 증거 검증"** 체크리스트로 1차 표기, %는 파생·부가로 강등 — 이러면 "코드로 계산"이 실제로 참인 영역만 전면에 남는다. (2) 본격 파이프라인 전에 **Wizard-of-Oz**(사람이 백엔드 흉내)로 5~10개 실프로젝트에 매핑을 돌려 "승인만 해도 되는 정확도"가 나오는지 검증하고, **제품 성립 임계 정확도선을 문서에 못박기.** (3) 의미비교는 임베딩 코사인으로 근접쌍 prefilter → LLM은 근접쌍만 판정 → 결과를 `(itemHashA, itemHashB, modelVersion)` 키로 캐시 + `temp=0`으로 재현성 확보.

### 11.5 제품 · 전략 리스크

- **"범용" 포지셔닝이 초기 wedge를 죽인다** — 대학생~PM~연구자 전부를 겨냥하지만(§6), 증거가 객관적인 곳은 커밋·테스트·실행화면이 있는 **SW뿐**이다. 발표·기획의 "완료조건/증거"는 주관적이라 매핑이 거의 불가능. → **MVP를 Git 기반 SW 단일 버티컬(솔로 개발자/인디해커)로 좁히고 "범용"을 마케팅에서 내린다.**
- **대체재 위협** — "지금 상태 뭐임/뭐 바뀜/근거 파일 어디"는 이미 Claude Projects·Cursor가 레포를 읽어 무설치·대화형으로 답한다. LINK의 유일한 추가가치는 진척도%인데 그게 가장 안 믿기는 기능. → 방어선을 **"지속 관찰로 쌓이는 변경 이력·SUPERSEDED 감사 궤적"** 처럼 세션형 챗봇이 못 하는 곳으로 재정의.
- **해자가 얕다** — Evidence-based Progress는 개념일 뿐, Linear/Jira가 "완료조건+증거링크" 필드를 붙이면 모방된다. → **사용자 승인·수정(§9.9)을 추출·매핑 모델의 학습 신호로 되먹여** "쓸수록 정확해지는" 복리 우위(데이터 루프)를 유일한 장기 방어선으로 설계.
- **가격·GTM·배포형태 공백** — 로컬퍼스트(에이전트+Ollama)인지 클라우드 SaaS(pgvector/Redis)인지조차 미정. "누가 왜 돈을 내는가" 한 문단이 없다.
- **콜드스타트** — 변화 구동 모델이라, 이미 성숙한 대형 레포/문서 폴더를 처음 연결하면 관측할 "변화"가 없어 첫 화면이 텅 비거나 0%. → 연결 시 1회 전체 백필(요약·구조 추출) 경로를 §5.6 예외로 명시.

### 11.6 권장 — MVP 재정의 & 첫 2주 수직 슬라이스

현재 §17 "MVP"는 6개 서브시스템 전부를 담은 사실상 **v1 전체**다(딜리버리·프로덕트 렌즈 지적). 핵심 베팅(증거기반 진척)이 Phase4까지 미검증인데, 검증에 불필요한 상주 에이전트를 초기 롱폴로 껴안아 속도를 죽인다.

**제안 — 첫 2주에 이 수직 슬라이스로 "자동 변화 감지"를 실증**:

```text
1. Project 생성 + Source/SourceVersion/ChangeEvent 최소 스키마
2. 프론트 "폴더 선택 + 지금 동기화" 버튼
   ↳ 상주 에이전트 대신 브라우저 File System Access API 또는 수동 스캔
3. 파일 읽기 → SHA-256 해시 → 이전 버전과 비교 (동일 해시 종료)
4. 텍스트 Diff 생성 + ChangeEvent 저장
5. 규칙 기반 변경 분류(추가/수정/삭제)만 — LLM 없음
6. CHANGES 화면에 변경 전/후·파일·일시 한 줄 렌더
```

- **상주 로컬 에이전트를 MVP에서 제거** → 온디맨드 스캔으로 대체. 가설 검증에 필요한 건 "지속 감시"가 아니라 "변경분을 한 번 받아 CHANGES를 그리는" 루프뿐. 배포·설치·자동업데이트·페어링 인증이라는 별도 제품을 초기에 안 짊어짐.
- **MAP · 의미형 충돌탐지 · ASK · Git 통합 · 6종 템플릿을 MVP 밖으로 명시 절단.** 충돌탐지는 "동일 필드 두 값"류 **규칙 기반 구조적 충돌만** 지원한다고 범위를 좁혀 선언(§9.6 예시 중 "마감일 두 날짜"만).
- **자기 문서를 첫 테스트 프로젝트로 도그푸딩** → 데모 데이터·정확도 골든셋·개발 피드백이 한 번에 해결.
- **21장 지표에 합격선·측정 시점 부여** — Phase4 진입 전 손라벨 골든셋(한 프로젝트, 20~30 변경) + 각 지표 최소 합격선을 사전 선언. 미달 기능은 자동판정을 끄고 규칙/수동으로 강등하는 게이트를 델리버리 규율로.
