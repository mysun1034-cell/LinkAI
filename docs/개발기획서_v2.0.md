## 1. 프로젝트 개요

### 1.1 프로젝트명
LINK AI

### 1.2 제품 정의
LINK AI는 사용자가 프로젝트를 생성하고 대화, 문서, 작업 디렉터리, Git 저장소를 연결하면 시스템이 작업 변화를 자동으로 감지하고 분석하여 프로젝트의 현재 상태, 변경사항, 실제 진척도와 다음 행동을 시각적으로 보여주는 범용 AI 생산성 플랫폼이다.

### 1.3 한 문장 정의
사용자가 기록을 직접 정리하는 도구가 아니라, 시스템이 먼저 정리하고 사용자는 중요한 결과만 확인·수정·승인하는 도구

### 1.4 제품 포지셔닝
LINK AI는 다음 제품이 아니다.

- 단순 메모 애플리케이션
- 문서 저장소
- 파일 탐색기
- 전체 지식 그래프 뷰어
- 일반 RAG 챗봇
- 단순 프로젝트 관리 프로그램
- 파일 수정 횟수만 집계하는 활동 분석기

LINK AI는 다음을 목표로 한다.

> 프로젝트의 작업 증거를 바탕으로 현재 상태와 실제 진척을 재구성하고, 변화와 관계를 시각화하며, 자연어로 탐색하고 행동까지 연결하는 **Project Intelligence Platform**

## 2. 개발 배경

사용자는 하나의 프로젝트를 수행하면서 여러 도구를 동시에 사용한다.

- 아이디어와 질의응답: ChatGPT, Claude 등
- 팀 커뮤니케이션: 메신저, 이메일
- 문서 작성: 워드프로세서, 노션, 마크다운
- 파일 관리: 로컬 디렉터리, 클라우드 드라이브
- 개발 작업: Git, IDE
- 업무 관리: 체크리스트, 칸반보드, 캘린더

이 과정에서 프로젝트 정보는 여러 채널에 분산되고, 과거 계획과 최신 결정이 혼재한다. 생산성 도구를 사용하면서도 사용자는 다시 폴더를 만들고, 태그를 붙이고, 링크를 연결하며, 최신 문서를 직접 구분해야 한다.

결과적으로 정보는 많지만 다음 질문에 즉시 답하기 어렵다.

- 프로젝트는 지금 어떤 상태인가?
- 최근 무엇이 바뀌었는가?
- 어떤 결정이 현재 유효한가?
- 실제로 어디까지 완료되었는가?
- 무엇이 프로젝트 진행을 막고 있는가?
- 지금 가장 먼저 해야 할 일은 무엇인가?
- 이 판단의 근거가 되는 파일과 기록은 무엇인가?

LINK AI는 이 문제를 **현재 상태 중심의 자동 구조화와 시각화**로 해결한다.

## 3. 핵심 문제 정의

- **3.1 정보 분산** — 같은 프로젝트에 관한 대화, 문서, 파일, 업무 기록이 서로 다른 서비스와 디렉터리에 존재한다.
- **3.2 오래된 정보 혼재** — 초기 계획, 중간 변경안, 최신 결정이 모두 검색되므로 현재 유효한 내용을 판별하기 어렵다.
- **3.3 생산성 도구의 역설** — 정보를 정리하기 위해 사용자가 태그, 폴더, 링크, 노드 관계를 계속 관리해야 한다.
- **3.4 활동량과 진척도의 혼동** — 파일 수정 수, 커밋 수, 작업 시간은 활동량을 보여주지만 실제 완료 정도를 증명하지는 못한다.
- **3.5 프로젝트 가시성 부족** — 자료는 쌓이지만 현재 상태, 변경 원인, 병목, 우선순위가 한 화면에 나타나지 않는다.
- **3.6 AI 비용과 의존성** — 모든 파일과 이벤트를 외부 LLM으로 분석하면 비용, 속도, 보안, 장애 의존성이 커진다.

## 4. 제품 목표

- **4.1 1차 목표** — 사용자가 LINK AI에서 프로젝트를 생성하고 작업 디렉터리를 연결하면, 변경사항이 자동으로 감지되어 NOW, MAP, CHANGES 화면에 반영되는 MVP를 구축한다.
- **4.2 중기 목표** — 파일과 작업 증거를 마일스톤 완료 조건에 연결하여 신뢰할 수 있는 실제 진척도를 계산한다.
- **4.3 장기 목표** — 프로젝트의 결정, 변화, 진척, 위험과 다음 행동을 이해하고 실행을 보조하는 Project Intelligence Platform으로 확장한다.

## 5. 핵심 설계 원칙

### 5.1 시스템이 먼저 정리한다
사용자는 태그와 링크를 계속 입력하지 않는다. 시스템이 자동으로 다음 항목을 추출하고 연결한다: 프로젝트, 목표, 결과물, 마일스톤, 업무, 결정, 사람, 일정, 파일, 변경사항, 증거, 위험, 질문.

### 5.2 온톨로지는 내부 구조로 사용한다
프로젝트 객체와 관계 구조는 시스템 내부에서 관리한다. 사용자는 복잡한 온톨로지나 전체 노드 그래프가 아니라 업무에 필요한 화면을 본다.

### 5.3 시각화는 행동과 연결한다
시각화는 정보를 구경하는 기능이 아니다. 사용자는 시각화 화면에서 다음 행동을 실행할 수 있어야 한다: 현재 정보로 승인, 검토 요청, 업무 생성, 담당자 지정, 마감일 설정, 관련 파일 열기, 변경 영향 확인.

### 5.4 규칙·캐시·데이터베이스 우선
파일 감지, 해시, 버전 비교, 상태 계산, 진척도 계산은 일반 코드가 담당한다. LLM은 자유로운 문장 해석과 의미적 비교가 필요한 경우에만 사용한다.

### 5.5 원본과 AI 해석 결과를 분리한다
원본 문서와 대화는 수정하지 않고 보존한다. AI가 추출한 정보는 후보로 저장하고, 검증 후 현재 상태에 반영한다.

### 5.6 전체 재분석을 금지한다
디렉터리 전체를 매번 LLM에 전달하지 않는다. 이전 버전과 비교한 변경분만 분석한다.

## 7. 핵심 개념

### 7.1 Project Digital Twin
현실 프로젝트에서 발생하는 목표, 결정, 파일, 작업, 변경, 진척 상태를 LINK AI 내부에 동기화한 프로젝트 모델이다.

```text
현실 프로젝트
- 대화
- 문서
- 코드
- 파일
- Git 커밋
- 체크리스트
- 작업 결과물

        ↓ 동기화

LINK AI Project Twin
- 현재 목표
- 현재 결정
- 결과물 상태
- 마일스톤
- 변경 이력
- 진척도
- 위험
- 다음 행동
```

### 7.2 Current State
프로젝트에서 현재 유효한 정보와 상태를 의미한다. 과거 정보는 삭제하지 않고 변경 이력으로 보존하되, 기본 화면에서는 현재 유효한 정보를 우선 표시한다.

### 7.3 Evidence-based Progress
진척도를 활동량이 아니라 완료 조건과 검증된 증거를 기반으로 계산하는 방식이다.

```text
마일스톤
→ 완료 조건
→ 증거
→ 검증
→ 진척도 반영
```

### 7.4 Activity와 Progress

- **Activity** — 수정 파일 수, 커밋 수, 문서 추가 수, 작업 시간, 최근 활동 빈도
- **Progress** — 완료 조건 충족, 검증된 결과물, 통과한 테스트, 승인된 결정, 실제 마일스톤 달성

활동량은 높지만 진척도가 낮을 수 있으며, 두 값을 반드시 분리해 표시한다.

### 7.5 Freshness와 Conflict
정보에는 다음 상태를 부여한다.

- `CURRENT`: 현재 유효
- `PENDING`: 확인 대기
- `CONFLICT`: 다른 정보와 충돌
- `SUPERSEDED`: 새로운 정보로 대체
- `STALE`: 장기간 확인되지 않음
- `ARCHIVED`: 참고용 보관
- `INVALID`: 잘못된 정보

## 8. 사용자 이용 흐름

1. **8.1 프로젝트 생성** — 프로젝트명과 간단한 설명을 입력한다.
2. **8.2 목표와 완료 기준 설정** — 프로젝트 목표, 최종 결과물, 마감일, 주요 마일스톤, 마일스톤별 완료 조건을 설정한다. LLM은 초안을 제안할 수 있으나 사용자가 최종 승인한다.
3. **8.3 디렉터리 연결** — 프로젝트와 관련된 로컬 폴더 또는 Git 저장소를 연결한다.
4. **8.4 최초 스냅샷 생성** — 시스템은 디렉터리 구조, 대상 파일, 콘텐츠 해시, 수정시간을 저장한다.
5. **8.5 변경 감지** — 로컬 에이전트가 파일 생성, 수정, 삭제, 이름 변경과 Git 이벤트를 감지한다.
6. **8.6 Diff 생성** — 이전 버전과 신규 버전의 차이를 계산한다.
7. **8.7 의미 분석** — 규칙으로 처리 가능한 변경은 코드로 분석하고, 애매한 변경만 LLM에 전달한다.
8. **8.8 상태 갱신** — 분석 결과를 현재 상태, 변경 이력, 업무, 마일스톤 증거에 반영한다.
9. **8.9 사용자 확인** — 확신이 낮거나 중요한 변경은 검토함에서 사용자가 승인·수정한다.
10. **8.10 시각화 갱신** — NOW, MAP, CHANGES, TASKS 화면이 자동으로 갱신된다.

## 9. 핵심 기능 요구사항

### 9.1 프로젝트 생성 및 템플릿
- **기능**: 프로젝트 생성/설명/유형 선택, 목표·결과물·마감일 설정, 마일스톤·완료 조건 생성
- **프로젝트 템플릿**: 소프트웨어 개발, 발표 준비, 연구·논문, 사업 기획, 콘텐츠 제작, 자유 프로젝트
- **완료 기준**: 사용자가 새 프로젝트를 생성하고, 템플릿 선택 시 기본 마일스톤 초안이 생성되며, 사용자가 항목을 수정·승인할 수 있다.

### 9.2 대화·문서·파일 수집
- **입력 방식**: 텍스트 복사·붙여넣기, TXT/Markdown/PDF/DOCX/JSON 대화 내보내기 파일 업로드, 로컬 디렉터리 연결, Git 저장소 연결
- **원칙**: 원본 파일 보존, 파일 해시 생성, 동일 파일 재분석 금지, 민감정보 파일 기본 제외

### 9.3 로컬 디렉터리 동기화
웹 브라우저만으로 지속적인 로컬 폴더 감시가 어려우므로 별도의 로컬 에이전트를 사용한다.

- **로컬 에이전트 역할**: 사용자 지정 디렉터리 감시, 파일 생성·수정·삭제·이동·이름 변경 감지, 해시 계산, 변경분 추출, 민감정보 필터링, 서버로 이벤트 전달
- **기본 무시 항목**:

```text
.git/objects/
node_modules/
target/
build/
dist/
.venv/
__pycache__/
*.class
*.exe
*.dll
*.zip
*.mp4
.env
secrets/
```

- **변경 이벤트 중복 처리**: 파일 저장 시 다수 이벤트가 발생할 수 있으므로 debounce를 적용한다. 권장 범위: 3~10초

### 9.4 변경 감지 및 Diff
- **처리 대상**: 텍스트 변경, 문단 추가·삭제, 코드 변경, 체크박스 상태 변경, 파일 생성·삭제·이름 변경, Git 커밋, 결과물 생성
- **처리 방식**:

```text
변경 이벤트 발생
→ 콘텐츠 해시 확인
→ 동일 해시이면 종료
→ 이전 버전 조회
→ Diff 생성
→ 규칙 분석
→ 필요 시 LLM 분석
→ 상태 반영
```

### 9.5 Current State 관리
- **기능**: 현재 유효한 목표·결정·일정, 진행 중 업무, 최신 결과물, 미해결 질문, 현재 위험, 최근 변경사항
- **처리 원칙**: 새 정보가 기존 정보를 대체하면 기존 정보는 `SUPERSEDED`로 보존한다. 충돌이 확실하지 않으면 `CONFLICT` 또는 `PENDING`으로 처리한다. 중요한 상태 변경은 사용자 승인을 요구한다.

### 9.6 충돌 및 최신성 관리
- **충돌 예시**: 마감일이 두 날짜로 기록됨, 담당자가 서로 다르게 기록됨, 문서별 프로젝트 범위가 다름, 최신 기획서와 업무 목록이 불일치
- **화면 제공 정보**: 기존 정보, 신규 정보, 각각의 출처, 작성일, 신뢰도, 추천 처리 방식
- **사용자 행동**: 기존 정보 유지, 신규 정보 승인, 둘 다 유효, 추가 검토 요청

### 9.7 Evidence-based Progress
- **마일스톤 예시**: `대시보드 구현`
- **완료 조건 예시**: NOW/MAP/CHANGES 화면 구현, 테스트 데이터 표시, 주요 클릭 동작 확인
- **증거 예시**: 소스 코드 파일, 실행 화면, 테스트 로그, Git 커밋, 검토 문서, 사용자 승인
- **상태 단계**: 시작 전 0% / 진행 중 1~59% / 구현 후보 약 60% / 검증 완료 약 90% / 승인 완료 100%
- 실제 백분율은 조건별 가중치와 검증 결과를 코드로 계산한다. **LLM이 임의로 백분율을 결정하지 않는다.**

### 9.8 시각화 화면

- **NOW** — 현재 가장 중요한 상태: 진행 중 프로젝트, 최근 변경, 확인 필요, 마감 임박, 지연 업무, 다음 행동, 최근 활동 추이
- **MAP** — 현재 프로젝트와 관련된 관계만 표시(프로젝트/목표/결정/업무/사람/문서/일정/위험/결과물). 전체 그래프를 기본 노출하지 않고 최대 20~30개 노드만 표시한다.
- **CHANGES** — 시간 순서로 변경사항 표시: 변경 전/후, 변경 일시, 변경 파일, 변경 이유, 영향을 받는 업무, 승인 상태
- **TASKS** — 업무명, 담당자, 상태, 마감일, 관련 결정, 관련 결과물, 지연 여부
- **ASK** — 사용자가 자연어로 화면을 탐색한다. 예: "이번 주 변경된 결정만 보여줘", "마감이 임박한 업무를 보여줘", "이 결정에 영향을 받는 파일을 찾아줘", "지난주 변경된 API를 알려줘", "충돌하는 일정만 보여줘"

### 9.9 Action 기능
- **가능한 행동**: 정보 승인, 충돌 해결, 업무 생성, 담당자 지정, 마감일 변경, 검토 요청, 관련 파일 열기, 보고서 생성
- **권한 원칙**: 조회·추천은 자동 / 업무 후보 생성은 자동 제안 / 일정 변경은 사용자 승인 / 파일 수정은 미리보기 후 승인 / 삭제·대규모 변경은 명시적 확인 필수

## 10. LLM 역할과 호출 정책

### 10.1 LLM 없이 처리하는 기능
파일 변경 감지, 콘텐츠 해시, 버전 비교, 날짜 파싱, 파일 분류, 상태 머신, 체크리스트 집계, 업무 완료 계산, 마일스톤 진척도 계산, 화면 필터링, 대시보드 집계, 권한 처리, 캐시 처리.

### 10.2 LLM이 담당하는 기능
자유형 회의록 구조화, 사실·결정·업무 후보 추출, 지시어 해석, 의미적으로 같은 업무 판단, 의미적 충돌 분석, 변경 영향 설명, 자연어 질의 해석, 화면 명령 생성, 사용자 친화적 설명 생성.

### 10.3 LLM Routing

```text
1. Cache
2. Rule Engine
3. Local Model
4. External API
5. Human Review
```

외부 API는 최종 fallback으로만 사용한다.

## 19. MVP 완료 기준

다음 조건을 **모두** 충족해야 MVP 완료로 판단한다.

1. 사용자가 프로젝트를 생성할 수 있다.
2. 목표, 결과물, 마일스톤과 완료 조건을 등록할 수 있다.
3. 로컬 디렉터리를 프로젝트에 연결할 수 있다.
4. 파일 생성·수정·삭제 이벤트가 감지된다.
5. 동일 파일은 재분석하지 않는다.
6. 변경된 텍스트에 대한 Diff를 생성한다.
7. 규칙 또는 LLM을 통해 변경 요약을 생성한다.
8. 프로젝트 Current State를 저장한다.
9. 충돌 정보가 검토함에 표시된다.
10. 파일이나 결과물을 완료 조건의 증거로 연결할 수 있다.
11. 근거 기반으로 마일스톤 진척도가 계산된다.
12. NOW 화면에서 현재 상태와 다음 행동을 볼 수 있다.
13. MAP 화면에서 주요 관계를 볼 수 있다.
14. CHANGES 화면에서 변경 이력을 볼 수 있다.
15. ASK에서 기본 질문에 근거와 함께 답할 수 있다.
16. 사용자가 결과를 승인·수정·거절할 수 있다.

## 25. 최종 결론

LINK AI의 경쟁력은 많은 AI 기능을 제공하는 데 있지 않다. 핵심은 다음 세 가지다.

1. 사용자가 작업하는 **실제 디렉터리와 기록에서 변화를 자동으로 감지**한다.
2. 활동량이 아닌 **완료 조건과 증거를 바탕으로 실제 진척도를 계산**한다.
3. **현재 상태와 변화, 병목, 다음 행동을 시각적으로 보여준다.**

따라서 LINK AI는 다음과 같이 정의한다.

> LINK AI는 프로젝트의 작업 흔적을 지속적으로 관찰하고, 변경의 의미와 실제 진척도를 재구성하며, 사용자가 지금 봐야 할 상태와 다음 행동을 시각화하는 **범용 Project Intelligence Platform**이다.
