## 결정 1 — 배포 형태: 로컬퍼스트(재정의)

**제안**: (A) **로컬퍼스트**를 출시 포지션으로 확정. 단 "사용자가 서버 스택을 운영하는 자가호스팅"이 아니라 **"로컬 설치형 단일 앱 — raw_content가 기기를 절대 떠나지 않는다"** 로 재정의.

- 추론은 MVP에선 **BYO-key**(사용자 자기 API 키로 외부 호출) + **로컬 Ollama** 옵션.
- 데이터루프 해자는 **'비원문 구조화 교정 텔레메트리 옵트인'** 으로 별도 확보.
- 얇은 클라우드 동기화(제약형 C, 파생·비원문 상태만)는 PMF 이후 재검토.

**왜**: 세 근거가 한 점으로 수렴한다.
1. 타깃(자기 코드가 경쟁우위인 프라이버시 민감 솔로 개발자)에겐 원문 서버 전송 자체가 채택 장벽.
2. 해자가 얕고 대체재(Claude Projects·Cursor)가 강함 → 방어 가능한 유일 차별점은 "코드가 기기를 안 떠난다"인데 **B/C는 이걸 스스로 폐기**.
3. 6렌즈 🔴2(raw_content 서버저장·"Local Only" 오명명·신뢰경계)의 뿌리가 전부 "원문이 서버에 산다"인데, **A는 그 원인을 제거**해 오명명·신뢰경계·PIPA 부담을 동시에 해소.

| 선택지 | 핵심 장점 | 핵심 단점 | 적합도 |
| --- | --- | --- | --- |
| **A. 로컬퍼스트(재정의)** | Local Only가 진짜가 됨·프라이버시 차별·PIPA 이탈·COGS 회사부담 0 | 설치 마찰·중앙 데이터루프 상실·패키징 비용·**기획 서버스택(Spring/PG/Redis)과 상충** | **최상**(단 무마찰 성장엔 약함) |
| B. 클라우드 SaaS | 마찰 최소·빠른 반복·중앙 데이터 해자 | raw_content 서버 상주(최대 지적 방치)·PIPA 처리자 전면 인수·차별점 폐기 | 타깃과 정면 충돌 |
| C. 하이브리드(기획안) | UX·로컬캡처 절충·원 기획스택 정합 | 복잡도 최고(3대 리스크가 전부 여기)·오명명 지속·PIPA 부담 | MVP 속도 최악 |

**✅ 확인된 사실**
- **File System Access API의 로컬 디스크 디렉터리 피커는 Chromium 계열(Chrome/Edge/Opera)만 지원.** Firefox·Safari는 OPFS만, 로컬 디스크 피커 미지원 → '브라우저 온디맨드 스캔' MVP는 Chromium 전용 제약. *(caniuse.com/native-filesystem-api, MDN — 본 세션 지식과 일치)*
- **Spring AI는 Ollama 챗·임베딩 자동구성 + pgvector 연동 제공** → 로컬 LLM/임베딩 경로 기술적 성립. *(docs.spring.io/spring-ai)*

**⚠️ 미확인·확인 필요**
- 타깃이 설치 마찰을 감수하고 로컬 앱을 채택하는가 (→ 사용자 인터뷰 필요, 현재는 전략적 추정)
- 솔로 빌더 기준 데스크톱/임베디드 패키징·크로스플랫폼 배포 실공수 (미산출)
- 소스/문서 파일 내용이 PIPA상 '개인정보'인지 (→ 서버 보유 여부의 법적 트리거, 변호사 확인)

**파급(cascade)**: A는 스택을 바꾼다 — **임베디드(SQLite+로컬 벡터)로 재설계하거나 Spring 로컬 번들**로 가야 하고 서버 Postgres/Redis는 MVP 스코프아웃. 캐시 키는 워크스페이스가 아니라 **리포/브랜치 단위**가 자연스러워져 6렌즈 🔴1(stateVersion)도 완화. 로컬 실행이라 PIPA 처리자 노출 급감(결정 5의 절반을 설계로 선제거).

**액션**
- 🔴 now: A(로컬퍼스트) 출시 포지션 확정 + `raw_content는 기기를 떠나지 않는다`를 하드 불변식으로 명문화(서버 원문저장 삭제) + 추론 기본경로=BYO-key
- 🔴 now: 스택 재정의 결정 — 임베디드(SQLite+로컬벡터) vs Spring 로컬 번들, 공수 견적
- next: 타깃 축 확정 — Chromium 전용(브라우저 스캔) vs 데스크톱 앱 택1
- later: 비원문 교정 텔레메트리(옵트인) 설계 + PMF 후 얇은 동기화 재검토 조건

## 결정 2 — 첫 타깃: Git 솔로 개발자 + SUPERSEDED 웨지

**제안**: '범용' 폐기하고 **Git 기반 SW 솔로 개발자/인디해커** 단일 버티컬로. aha 웨지 = **`SUPERSEDED 감사궤적`**(지금 유효한 결정 vs 폐기된 결정의 시간축)을 주역으로, **커밋/테스트/CI 완료조건 체크리스트**를 신뢰 백본으로 결합.

> 첫 5분 aha = *"연결하니 내 리포의 '지금 유효한 결정 vs 폐기된 결정' 타임라인 + green CI로 검증된 완료항목이 자동으로 뜬다."*

**왜**: 이 버티컬에서만 '증거→완료조건'이 커밋·머지 PR·테스트·CI라는 **기계판정 가능 신호**로 환원되어 제품 최약점(의미판정)이 부분 무력화된다. 그리고 대체재(Cursor·Copilot·Claude Projects)는 모두 '현재 스냅샷·현재 세션' 도구 — **지속관찰·시간축 변경이력·SUPERSEDED 궤적을 코어로 가진 곳이 없다.** 이게 6렌즈의 '해자 얕음→교정 데이터루프가 유일 방어선'과 맞물린다(SUPERSEDED 오판을 교정할 때마다 라벨이 쌓임).

**✅ 확인된 사실 — 대체재 가격(모두 개인 $10~20 구간에 밀집)**

| 제품 | 가격 | 출처 |
| --- | --- | --- |
| Cursor Pro | $20/월(연 $16/월) | cursor.com/pricing |
| GitHub Copilot Pro | $10/월, **2026-06-01부터 전 플랜 사용량 기반(AI Credits)** 전환·Pro+ $39·Max $100 | github.blog |
| Claude Pro | $20/월($17 연). Projects는 무료 5개/Pro 무제한 | claude.com/pricing |
| Linear | Basic $10·Business $16/user·월 | linear.app/pricing |

→ 대체재가 전부 $10~20이라 **LINK도 그 앵커에 갇힘**. 제안 과금 = 초기 단일 정액 **$9~15/월**(무료=로컬only·1리포), 이후 '활성 리포 수' 전환 검토.

**⚠️ 미확인·확인 필요**
- 솔로 개발자의 '컨텍스트·문서 유실' 통증이 실제 지불로 이어지는가 (→ 6~10명 문제 인터뷰)
- SUPERSEDED(결정 대체) 판정을 로컬 소형 LLM이 한국어로 신뢰성 있게 하는가 (→ 결정 4 bake-off)
- GitHub Copilot Workspace 제품 정식 폐기 여부 (githubnext 프리뷰 상태 추정)

**파급**: 캐시는 '리포(가능하면 브랜치) 단위'가 자연 경계(결정1과 정합). 증거를 Git 객관신호로 제한 → LLM 의존도 구조적으로 낮아짐 → 로컬 모델·GPU 요구 완화. 과금 $9~15 상한 → AI 원가예산 극소 → 외부 LLM 최후 fallback 원칙이 경제적으로 강제.

**액션**
- 🔴 now: MVP 웨지 확정('SUPERSEDED 타임라인 + green CI 완료항목'을 첫 화면), 진척도% 체크리스트 강등 명문화
- 🔴 now: Git 솔로 개발자 6~10명 문제 인터뷰(통증 크기·$9~15 지불의사)
- now: 기계추출 가능 증거 신호 목록(머지 PR·close 이슈·테스트·CI·릴리스)과 LLM 의미판정 필요 지점(SUPERSEDED 동치) 분리 명세
- next: 경쟁 포지셔닝 1장(LINK='지속관찰·SUPERSEDED 궤적' vs 대체재='현재 스냅샷')

## 결정 3 — 진척도 표현: 체크리스트 우선 혼합

**제안**: 헤드라인 = **"완료조건 5/8 검증 · 2건 추정 · 1건 충돌"** 형태의 상태 분해 문장 + 목표별 체크리스트. **bare 정밀 %(63.4%)는 금지.** %는 체크리스트에서 결정론적으로 파생한 거친·버킷화 보조 지표로만(`Σ검증weight/Σweight`, 반올림·밴드). 모든 진척 숫자에 **confidence 티어(확정/추정/충돌/부족)** 동반.

**왜**:
- 단일 %는 **단일실패점** — 한 조건 오판이 전체 숫자를 오염(레드팀 확정 지적). 체크리스트는 오판이 해당 행에 국소화(blast radius 격리).
- 이 제품의 최약점(증거→조건 의미판정)을 %가 숨겨버림 → false precision은 추측금지 정신 위반.
- 해자(교정 데이터루프)는 **'매핑=제안(PENDING)·되돌리기 가능'** UX가 있어야 라벨을 만든다 → 체크리스트 구조가 그 훅.
- weight 기본=**균등**(추측금지의 정직한 기본). 비균등은 사용자 명시 결정 또는 '미확정' 태그가 붙은 AI 제안으로만.

**파급(중요)**: 이 결정은 **데이터 모델을 강제한다** — 완료조건·증거·매핑을 각각 **상태머신 가진 1급 엔티티**(미검증/증거약/검증됨/충돌/무효 + weight + confidence + provenance[판정자·시각·freshness])로 승격, **이벤트 소싱 전제**. 증거 드릴다운은 원문 전체가 아니라 **로컬 참조 포인터(경로·해시·오프셋)** 로 — 결정 1·5(원문 서버 미저장)와 정합.

**액션**
- 🔴 now: 완료조건·증거·매핑을 1급 엔티티로 정의(상태머신·weight·confidence·provenance, 이벤트소싱)
- 🔴 now: 헤드라인 지표 규격(bare % 금지, 상태분해 문장 + confidence 티어, %는 파생·버킷화, 파생식 드릴다운 노출)
- 🔴 now: weight 기본=균등 고정 + 사용자 조정·로깅, AI 제안은 '미확정' 태그로만 무적용
- next: 매핑 되돌리기 UX(기본 PENDING·승인/반려/재배정·비파괴 undo·감사추적) + 교정 이벤트를 라벨 데이터로 적재(해자)

- ★ **HyperCLOVA X SEED = 상업 이용 무료(월 1000만 MAU 이하), 초과 시 NAVER 별도 라이선스.** Think 14B/32B 포함 상업친화 라이선스로 공개. *(AVING/Pickool 보도 + HuggingFace)* — 딥다이브의 medium confidence보다 오히려 확실해짐(MAU 상한 디테일 추가).
- **Mistral Small 3/3.2 24B = Apache-2.0**, 다국어(한국어 포함), 네이티브 function calling·JSON 설계목표, q4 양자화 시 단일 RTX4090/32GB Mac 구동. Ollama 공식 라이브러리 등재. *(mistral.ai/news, ollama.com/library/mistral-small3.2)*
- **Nemotron 3 Nano = NVIDIA Open Model License(상업 허용)**, MoE(활성 ~3B), 학습 15개 언어에 한국어 포함. *(HuggingFace nvidia/NVIDIA-Nemotron-3-Nano)*

**⚠️ 미확인·확인 필요**
- SEED Think 32B의 tool calling/JSON 구조화출력 신뢰성 (모델카드에 명시 없음 → 실측)
- **각 모델의 한국어 회의록 '결정 vs 논의' 판별·업무 추출 정확도(F1)와 Ollama 실지연/VRAM** — 공개 벤치는 대리지표일 뿐 **반드시 도메인 gold set으로 실측**(추측금지)
- Mac mini 소형 하드웨어에서 24~32B q4의 실사용 지연이 온디맨드 스캔 UX에 충분한지

**bake-off 프로토콜(제안)**
- **gold 평가셋**: 실제 한국어 회의록/문서 30~50건 사람 라벨(사실·결정·업무) + 목표 JSON 스키마 정의. 공개벤치로 대체 금지.
- **지표·합격선(초안)**: ①JSON 스키마 유효율 **≥98%(하드게이트)** ②추출 F1(결정·업무) **≥0.80** ③tool calling 성공률 **≥90%** ④지연 p50/p95 ⑤VRAM. *숫자는 실측 후 Chris 승인으로 확정.*
- **go/no-go**: 어떤 비중국 로컬 모델도 F1 합격선을 못 넘으면 그 태스크는 **External LLM로 라우팅**(Cache→Rule→Local→External 사다리 편입). 즉 bake-off는 '로컬 강제'가 아니라 **'어디까지 로컬, 어디서 External'의 경계선**을 정하는 일.

**파급**: 이 결정은 hardware 노선(Mac mini 소형 시작 → 24B q4 현실 상한)과 정합. EXAONE 탈락으로 '국산=EXAONE' 기본가정이 깨지고 **국산 앵커가 HyperCLOVA X SEED로 이동**. 로컬 모델은 '후보 추출'까지만, 최종 판정은 사용자 교정(결정 3 해자)으로.

## 결정 5 — PIPA: crypto-shredding + Local 기본 + 변호사 게이트

**제안**: **옵션 A(사용자/워크스페이스별 암호화키 + crypto-shredding)** 를 파기·삭제권 해소의 기본 아키텍처로 채택하되, **'변호사 확인'을 착수 전 필수 게이트**로. MVP 기본 모드 = **Local Only 고정**, 외부 LLM(국외이전)·민감정보 처리는 각각 **독립 opt-in 별도동의** 뒤에서만 활성화.

**핵심 충돌**: 제품 §7.2 **'과거 삭제 안 함(무삭제/append-only)'** 이 PIPA 제21조(파기의무)·제37조(처리정지+파기)·삭제요구권과 **정면 충돌**. 해소책 = 키 파기(crypto-shredding)를 파기 인정 경로로 — **단 PIPA가 이를 명시 인정하는지는 명문 규정 미확인 → 변호사 필수.**

| 선택지 | 장점 | 단점 | 적합 |
| --- | --- | --- | --- |
| **A. Crypto-shredding(키 파기)** | 무삭제 철학+파기의무 동시 충족·삭제요구 시 키 폐기 1회·GDPR/NIST Purge 등가 | **PIPA 제21조 인정 미확인**·키관리 공격면·백업 평문 잔존 위험 | 유일하게 철학 보존(단 확인 필요) |
| B. 물리삭제+tombstone | 제21조 논쟁 없음·키관리 없음 | Current State 재계산·이벤트소싱·SUPERSEDED 이력과 충돌(심장 훼손) | 규제 안전 최고, 제품가치 훼손 |
| C. 논리삭제 플래그만 | 구현 최단 | 원문 복구가능 잔존→**제21·36조 실질 미이행 위법 리스크**·오명명 지속 | **채택 금지 권고** |

**✅ 확인된 사실 — PIPA 조문 (모두 law.go.kr 국가법령정보센터)**
- **제28조의8(국외이전)**: 원칙 금지, 정보주체 **별도 동의** 시 예외. 5개 고지(①이전받는 자 ②목적·기간 ③항목 ④국가·일시·방법 ⑤거부 방법·효과), 변경 시 재동의. → **외부 해외 LLM API 전송이 국외이전에 해당할 수 있음.**
- **제23조(민감정보)**: 건강·사상·정치 견해 등 처리 원칙 금지, **다른 동의와 구분된 별도 명시동의** 또는 법령 근거만 예외. 처리 시 제29조 안전성 확보조치.
- **제21조(파기)**: 목적 달성·기간 경과 시 지체 없이, **복구·재생 불가하게** 파기.
- **제37조(처리정지)**: 정보주체 처리정지·동의철회 가능, 처리자는 지체없이 정지·파기 등 조치.
- **'안전성 확보조치 기준' 고시**: 접속기록 원칙 **1년 이상**(민감정보·고유식별정보 또는 5만명 이상 처리 시스템·기간통신사업자는 **2년 이상**) 보관, 월 1회 이상 점검.
- crypto-shredding은 NIST SP 800-88 Rev.1의 'Purge(암호소거)'로 분류, GDPR/EDPB 맥락에서 erasure 등가 통용 *(단 PIPA 직접 근거는 아래 미확인)*.

**⚠️ 미확인·확인 필요 (변호사/PIPC 게이트)**
- **crypto-shredding이 PIPA 제21조 '복구·재생 불가 파기'로 인정되는가** — GDPR/NIST 통용례는 있으나 PIPA 명문·유권해석 미확인
- 외부 LLM에 '전송·처리만 하고 미저장'도 제28조의8 국외'이전'인가, 처리위탁(제26조) 경로 가능한가
- LINK 데이터(코드·문서·Git)에 민감정보/고유식별정보 포함 개연성 → 접속기록 2년 트리거 여부
- CSAP가 판매 채널에서 실제 요구되는지 (CareLink 미해결 이슈·타깃이 민간이면 재판단)

**파급**: raw_content는 '사용자/워크스페이스별 키 암호화 + KMS 키 분리' 스키마로 → 결정 1의 '원문 서버 평문 적재=오명명' 직접 해소. Local 기본·외부 opt-in은 LLM 라우터의 External 단계에 '국외이전 동의 확인' 분기를 강제. **'변호사 확인' 게이트가 코드 착수의 크리티컬 패스**로 들어감.

**액션**
- 🔴 now: 법률자문 게이트 개설 — (a)crypto-shredding=PIPA 파기 인정? (b)외부 LLM 전송의 국외이전/위탁 분류 (c)민감정보 필터 실패 책임 — 로펌/PIPC 서면 질의
- 🔴 now: MVP 기본 모드 Local Only 고정 + 외부·민감 처리 독립 opt-in 별도동의 UI(제28조의8 5개 고지·제23조 별도동의 문구)
- next: raw_content 암호화+키 분리 스키마, 삭제요구·처리정지 시 '키 파기' 이벤트 정의(백업도 동일 키 종속)
- later: crypto-shredding 불인정 대비 폴백(삭제권 레코드 물리삭제+tombstone 하이브리드)
